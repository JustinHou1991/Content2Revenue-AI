# 架构决策记录 (Architecture Decision Records)

## ADR-001: 为什么选择Streamlit而非Next.js/Gradio

- **日期**: 2026-05-06
- **背景**: 1周MVP约束下需要最大化开发效率，开发者是Python基础水平
- **决策**: 使用Streamlit作为前端框架
- **理由**:
  - 学习曲线最低：写Python脚本即出Web页面
  - 多页面原生支持：`st.navigation` + `pages/` 目录
  - LLM流式输出原生支持：`st.write_stream()`
  - 数据展示组件丰富：`st.metric`, `st.tabs`, `st.expander`
  - 免费部署：Streamlit Cloud一键部署
- **后果**: 前端定制性受限，Phase 2可迁移至Next.js
- **替代方案**: Gradio（ML demo更优但数据应用较弱）、Next.js（更灵活但1周不够）

## ADR-002: 为什么使用SQLite而非PostgreSQL

- **日期**: 2026-05-06
- **背景**: MVP阶段需要轻量级数据持久化方案
- **决策**: 使用SQLite
- **理由**:
  - Python标准库自带`sqlite3`，零额外依赖
  - 单文件数据库，备份就是复制文件
  - 对于MVP（<10万条记录）性能完全够用
  - SQL语法通用，后续迁移到PostgreSQL成本低
- **后果**: 不支持高并发写入，不适合多用户同时写入场景
- **替代方案**: JSON文件（无查询能力）、PostgreSQL（运维成本高）

## ADR-003: 为什么不用litellm而直接用OpenAI SDK

- **日期**: 2026-05-06
- **背景**: 需要统一接口支持DeepSeek和通义千问
- **决策**: 使用OpenAI Python SDK + 简单封装
- **理由**:
  - DeepSeek和通义千问都原生支持OpenAI兼容协议
  - litellm的"统一接口"优势被削弱
  - litellm增加依赖复杂度，对Python基础用户不友好
  - 直接封装更透明，出问题时更容易调试
- **后果**: 切换到非OpenAI兼容模型时需要修改LLM Client
- **替代方案**: litellm（更通用但更复杂）

## ADR-004: 数据无关联时的核心策略

- **日期**: 2026-05-06
- **背景**: 销售线索数据和抖音脚本数据之间没有直接关联ID
- **决策**: 用AI建立语义层面的间接关联，项目价值从"归因分析"转向"智能推荐"
- **理由**:
  - 没有直接关联无法做传统归因分析
  - 用LLM分别提取内容画像和线索画像，在语义空间做匹配
  - "智能推荐"比"归因分析"对业务决策者更有直接价值
  - 不需要证明因果性，只需提供相关性推荐
- **后果**: 匹配精度依赖LLM提取质量，需要精心设计prompt
- **替代方案**: 强制要求数据关联（不可行）、放弃匹配功能（价值降低）

## ADR-005: JSON输出的三重保障机制

- **日期**: 2026-05-06
- **背景**: LLM输出JSON不稳定，解析失败会导致整个Pipeline中断
- **决策**: 实现三重保障机制
- **保障1**: API层 — `response_format={"type": "json_object"}`
- **保障2**: Prompt层 — 每个Prompt末尾包含明确的JSON Schema示例
- **保障3**: 后处理层 — 解析失败时自动重试 + LLM修复损坏JSON
- **理由**: 单一保障不够可靠，三重保障确保Pipeline稳定性

## ADR-006: 配置管理策略 — 三级优先级

- **日期**: 2026-05-06
- **背景**: API Key、模型参数等配置分散在代码各处，缺乏统一管理；不同部署环境（本地开发、Streamlit Cloud、私有部署）需要不同的配置来源
- **决策**: 实现三级配置优先级机制，由 `config.py` 统一管理
- **优先级**:
  1. **环境变量**（最高）: `DEEPSEEK_API_KEY`、`DASHSCOPE_API_KEY` 等，适合 CI/CD 和容器化部署
  2. **secrets.toml**: Streamlit Secrets 机制，适合 Streamlit Cloud 部署，敏感信息不入代码仓库
  3. **数据库设置表**（最低）: 应用内「系统设置」页面配置，支持运行时动态修改和持久化
- **理由**:
  - 环境变量是12-Factor App标准实践，Docker/K8s原生支持
  - secrets.toml 是 Streamlit 官方推荐的敏感信息管理方式
  - 数据库配置让非技术用户也能通过UI修改设置，降低使用门槛
  - 三级优先级确保开发/测试/生产环境平滑切换，无需改代码
- **后果**: 配置来源分散，调试时需要明确当前生效的配置来源；`config.py` 成为关键模块，需要充分测试
- **替代方案**: 全部使用环境变量（非技术用户不友好）、全部使用配置文件（敏感信息安全风险）、`.env` 文件（需要额外依赖 python-dotenv）

## ADR-007: 日志系统设计 — RotatingFileHandler + 模块级别别

- **日期**: 2026-05-06
- **背景**: MVP阶段使用 `print()` 输出调试信息，无法追踪问题；Streamlit 的 `st.write()` 不适合后端服务日志；需要持久化日志用于问题排查
- **决策**: 基于 Python 标准库 `logging` 模块，配合 `RotatingFileHandler` 实现自动轮转日志
- **设计要点**:
  - **日志文件**: `data/logs/c2r.log`，单文件最大 5MB，保留 3 个备份
  - **模块级别别**: 不同模块可设置不同日志级别（如 LLM Client 为 DEBUG，UI 层为 WARNING）
  - **统一入口**: `utils/logger.py` 提供 `get_logger(__name__)` 工厂函数
  - **日志格式**: `[时间] [级别] [模块名] 消息内容`，便于 grep 和分析
  - **双输出**: 同时输出到文件和控制台（开发时可见）
- **理由**:
  - Python 标准库自带，零额外依赖
  - `RotatingFileHandler` 自动管理日志文件大小，避免磁盘占满
  - 模块级别别控制避免日志噪音，生产环境可只保留 WARNING 以上
  - 统一入口确保所有模块日志格式一致
- **后果**: 日志文件需要纳入 `.gitignore`；长期运行可能需要日志收集方案（如 ELK）
- **替代方案**: `loguru`（更简洁但增加第三方依赖）、`st.logger`（Streamlit 内置但功能有限）、Sentry（适合生产监控但 MVP 阶段过重）

## ADR-008: 缓存策略 — 内容Hash + SQLite

- **日期**: 2026-05-06
- **背景**: 重复内容分析浪费API费用，相同脚本被多次分析时产生不必要的LLM调用成本
- **决策**: 基于内容MD5 hash的缓存系统，存储在SQLite数据库中
- **设计要点**:
  - **缓存键**: 内容MD5 hash，确保相同内容产生相同键
  - **存储位置**: `analysis_cache` 表，与业务数据同库管理
  - **过期策略**: TTL机制，支持按内容类型设置不同过期时间
  - **统计功能**: 缓存命中率、节省成本计算
- **理由**:
  - 零额外依赖，复用已有SQLite数据库
  - 自动过期机制，避免缓存数据过时
  - 统计方便，可直接查询数据库分析缓存效果
  - 与业务数据统一管理，备份和迁移简单
- **后果**: 缓存表会增长，需要定期清理过期数据；hash计算有微小性能开销
- **替代方案**: Redis（需要额外服务）、内存缓存（进程重启丢失）、文件缓存（管理复杂）

## ADR-009: A/B测试框架设计

- **日期**: 2026-05-06
- **背景**: 需要验证不同策略的效果，比较多种内容策略对转化率的影响
- **决策**: 独立的ABTestEngine服务，支持多版本策略生成
- **设计要点**:
  - **独立服务**: `services/ab_test_engine.py`，与核心业务解耦
  - **多版本生成**: 同一线索可生成多个策略变体
  - **随机分配**: 支持随机分组，确保测试公平性
  - **统计检验**: 内置统计显著性检验（t-test）
  - **数据存储**: `ab_tests` 表记录测试配置和结果
- **理由**:
  - 与核心业务解耦，不影响主流程稳定性
  - 支持统计显著性检验，确保结论可靠性
  - 可扩展支持多种测试类型（策略、Prompt、UI等）
  - 数据持久化，支持长期效果追踪
- **后果**: 增加了系统复杂度；需要足够的样本量才能得出有效结论
- **替代方案**: 外部A/B测试平台（增加依赖）、手动分组（容易出错）

## ADR-010: 成本追踪系统

- **日期**: 2026-05-06
- **背景**: 需要监控API使用成本，了解LLM调用的费用分布，寻找优化空间
- **决策**: 在LLMClient中集成成本计算，存储到数据库
- **设计要点**:
  - **集成位置**: `services/llm_client.py`，每次API调用后计算成本
  - **成本模型**: 基于token数量和模型单价实时计算
  - **数据存储**: `api_usage` 表记录每次调用的成本和元数据
  - **分析功能**: 按模块、按时间段、按模型统计成本
  - **可视化**: `ui/pages/cost_analytics.py` 提供成本分析页面
- **理由**:
  - 实时追踪，立即了解每次调用的成本
  - 历史分析，识别成本趋势和异常
  - 优化建议，基于数据发现高成本模块
  - 与业务数据关联，计算ROI
- **后果**: 每次API调用增加一次数据库写入；成本模型需要随供应商价格调整
- **替代方案**: 外部监控工具（如Langfuse，增加依赖）、日志分析（延迟高）
